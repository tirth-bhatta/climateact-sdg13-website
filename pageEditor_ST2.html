3. Splash Screen — Technical Description

🔗 View Splash Screen   ✅ Validation Evidence

HTML Structure

The Splash Screen does not use the shared team template, as specified in the coursework brief. It is a standalone page with a single <div> overlay structure. The page uses a <meta http-equiv="refresh"> tag to implement the automatic 4-second redirect to home.html, as taught in the lecture notes on automatic redirects in HTML links.[1]

Content includes the website title in an <h1> element, a mission statement paragraph, a list of group members, a CSS loader animation, a countdown paragraph, and a Skip Intro button.

CSS Techniques

All styles are embedded within a <style> block in the page's <head>. The background uses a full-screen image with background-size: cover and background-position: center to ensure it scales correctly on all screen sizes.[2]

A dark overlay (rgba(0,0,0,0.65)) is applied via a flex container to ensure text remains readable against the background image, meeting accessibility contrast requirements. Flexbox is used with justify-content: center and align-items: center to centre all content vertically and horizontally within the viewport.

CSS Loader Animation

A spinning CSS animation was created using @keyframes spin rotating a border-radius: 50% element from 0 to 360 degrees. This provides a visual loading indicator without using any image or GIF file, meeting the requirement for a CSS-only animation.

JavaScript Functionality

Two JavaScript features were implemented. First, a countdown timer uses setInterval() to decrement a variable and update the innerText of the countdown paragraph once per second. Second, the Skip Intro button calls a skipIntro() function that clears the interval timer, disables the meta refresh, and redirects immediately to home.html using window.location.href.

Design Rationale

The splash screen uses the team's agreed green colour palette and sans-serif typography to provide a visually seamless transition into the main website. The overlay approach was chosen over other layouts as it maximises the visual impact of the background image while maintaining text legibility.

4. Action Impact Simulator — Technical Description

🔗 View AIS Page   ✅ Validation Evidence

HTML Structure

The AIS page uses the shared team HTML template including the <header> with navigation bar, <main> section, and <footer> with page editor and validation links. The main content consists of six action cards, each structured as <div> elements with semantic content inside: a thumbnail <img>, an <h3> heading, a paragraph, and an impact label span.

A single <h1> is used as the page title ("SDG Action Impact Simulator"), following the correct heading hierarchy required for accessibility. The feedback area at the bottom of the page uses a <div id="feedback-area"> to display results dynamically.

CSS Techniques

Page-specific embedded CSS is defined inside a <style> block and contains more than five CSS rules as required. CSS Grid with grid-template-columns: repeat(auto-fit, minmax(240px, 1fr)) is used to create a responsive card layout.[3] The .selected class changes the card's border colour and background to provide a clear visual distinction when a card is selected.

JavaScript Functionality

The core interactivity is powered by a toggleCard() function. When a card is clicked, the function checks whether the card already has the selected CSS class. If it does, the class is removed and the card's point value is subtracted from the running total; if not, the class is added and the points are added. Conditional logic (if / else if / else) determines the impact level based on defined thresholds (Low: 0–7, Medium: 8–14, High: 15+).[4]

The background image of the page is changed using document.body.style.backgroundImage, controlled entirely by JavaScript based on the calculated impact level. This satisfies the requirement for JavaScript-controlled background image changes (not CSS tricks).

A resetAll() function iterates over all cards using querySelectorAll() and removes the selected class from each, resetting the total score to zero.

Accessibility

All card images include descriptive alt text. Cards have tabindex="0" and role="button" attributes, with a keyboard event listener for the Enter and Space keys, ensuring keyboard accessibility.[5]

Design Rationale

CSS Grid was chosen for the card layout because it automatically adapts to different screen widths, making the page usable on both desktop and mobile without writing separate breakpoints. The green colour scheme was selected to reinforce the climate/environmental theme and maintain consistency with the rest of the website.

5. Content Page — Technical Description

🔗 View Content Page   ✅ Validation Evidence

HTML Structure

The Content Page uses the shared team HTML template. The main content is divided into six clearly defined sections using <section> elements with unique id attributes, allowing for internal anchor linking. One <h1> is used for the page title, with <h2> and <h3> used for section and subsection headings following a logical hierarchy.

A secondary <nav> element provides an internal navigation menu at the top of the page, linking to each section using href="#section-id" anchor links. Five relevant images are included with descriptive alt attributes (exceeding the required minimum of four). A fixed "Go to Top" link (id="topBtn") is included using position: fixed CSS, requiring no JavaScript.

CSS Techniques

Page-specific embedded CSS handles the internal navigation styling, image borders and shadows, section spacing, and the fixed "Go to Top" button. Images use border-radius, box-shadow, and max-width: 100% to ensure they display responsively and with appropriate visual styling.

Design Rationale

The internal navigation list was placed at the top of the content area to help users on any device quickly understand the page structure and jump to relevant sections. This improves usability especially on long pages viewed on mobile devices where scrolling is the primary interaction.

6. Challenges & Lessons Learned

Challenge 1: JavaScript Toggle Logic

Initially, clicking a card multiple times caused the score to keep adding rather than toggling. I resolved this by checking card.classList.contains("selected") before deciding whether to add or subtract the points, and only then toggling the class.

Challenge 2: Background Image Changes

Setting the background image correctly using JavaScript required using the full url('path') syntax within style.backgroundImage. At first I only passed the file path without the url() wrapper, which did not work. I learned to always use the correct CSS value format even when setting styles via JavaScript.

Challenge 3: Meta Refresh and Skip Button

Making the Skip button cancel the meta refresh was not straightforward because the meta tag cannot be truly removed after page load. I solved this by changing its content attribute to a very large delay value, effectively disabling it, before immediately redirecting via window.location.href.

Challenge 4: Heading Hierarchy

During validation I discovered that the AIS page was using an <h2> as the main page title instead of an <h1>. This broke the heading hierarchy and accessibility requirements. I corrected this by changing the element to <h1> and updating the associated CSS selector from .ais-header h2 to .ais-header h1.

Lessons Learned

This project improved my understanding of the relationship between HTML, CSS, and JavaScript. I also learned the importance of planning the JavaScript logic on paper before coding it, and of testing interactivity across different browsers.

7. Accessibility Considerations

Accessibility was treated as a core requirement, not an afterthought, across all pages I implemented.

Alternative Text

Every image on my pages includes a meaningful alt attribute describing the image content, enabling screen readers to convey the image's purpose to visually impaired users.[5]

Semantic HTML

I used semantic elements including <header>, <nav>, <main>, <section>, and <footer> throughout. These elements communicate page structure to assistive technologies and improve navigation for screen reader users.

Heading Hierarchy

A single <h1> is used per page, with <h2> and <h3> in descending order. This logical hierarchy supports screen reader users in navigating page structure. The AIS page was specifically corrected to ensure its main title uses <h1> rather than <h2>.

Keyboard Navigation

Action cards on the AIS page are keyboard-accessible through tabindex="0" and keyboard event listeners. This ensures users who cannot use a mouse can still interact with all features.

Colour Contrast

I used dark green text on light backgrounds throughout, maintaining a high contrast ratio. The dark overlay on the splash screen ensures text remains legible against the background image in all lighting conditions.
