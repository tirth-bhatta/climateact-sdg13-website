📝 Reflection on Validation Process

Errors Encountered

  • Missing alt attributes: Initially, some images in the scrollable program cards section did not have proper alt text. This was resolved by adding descriptive alt attributes to all images.
  • Unclosed div tags: During initial validation, one div in the feedback form was not properly closed, causing nesting issues. Fixed by ensuring all tags are properly closed.
  • Duplicate IDs: The character counter element had a duplicate ID issue. Resolved by using unique IDs across the page.
  • ARIA roles on non-interactive elements: Initially added role="button" to program cards without proper keyboard handlers. Fixed by removing unnecessary ARIA roles and using proper semantic HTML with tabindex="0".

How They Were Resolved

  • Alt text: Added descriptive alt text to all images, including fallback placeholder images with onerror handlers.
  • Tag closure: Used a code linter to identify unclosed tags and fixed all nesting issues.
  • Unique IDs: Ensured all IDs are unique across the page; used separate IDs for different form elements.
  • ARIA improvements: Replaced role="button" with proper tabindex="0" for keyboard-accessible cards and used native <button> elements where appropriate.

What I Learned

  • Validating HTML early in development saves significant time compared to fixing issues at the end.
  • The W3C validator is an invaluable tool for catching structural mistakes that browsers silently correct but affect accessibility.
  • Semantic HTML is not just about validation — it significantly improves accessibility for screen reader users.
  • Proper heading hierarchy (h1 → h2 → h3) is crucial for both SEO and accessibility.
✅ Final Validation Status: Both pages (feedback.html and team.html) successfully passed W3C HTML validation with zero errors. Any warnings present are acceptable and do not affect compliance with web standards.
