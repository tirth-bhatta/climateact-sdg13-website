4. Reflection on Validation

Errors Encountered

During initial validation of ais.html, the validator flagged a missing alt attribute on one of the action card images. This was quickly corrected by adding a descriptive alt text string to the relevant <img> element.

A more significant issue on ais.html was the use of an <h2> element as the main page title instead of an <h1>. This broke the heading hierarchy and was flagged during accessibility review. It was corrected by changing the element to <h1> and updating the associated CSS selector accordingly.

On content_ST2.html, the validator initially flagged a heading hierarchy issue where an <h3> element was used before an <h2> within the same section. This was corrected by restructuring the headings into the correct logical order.

How Errors Were Resolved

I addressed each error systematically by reading the validator's line reference, locating the issue in my code editor, making the correction, and re-running validation until no errors remained. The process of iterative validation significantly improved my understanding of valid HTML structure.

Lessons Learned

Validation taught me that small oversights — such as a missing attribute, an out-of-order heading, or using the wrong heading level for a page title — can prevent a page from meeting web standards. Running validation early and often during development (rather than only at the end) makes fixing issues much quicker and less stressful. I also learned that warnings are informational and do not prevent successful validation.
