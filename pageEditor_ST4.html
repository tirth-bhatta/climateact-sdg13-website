User Profile Page — profile.html

View Live Page ↗ Validation Evidence ↗

HTML Structure

The page uses the shared template structure with <header>, <main>, and <footer>. Inside <main>, I used a <section> for the progress indicator (with role="progressbar" for accessibility), a <div class="control-row"> for action buttons, and a second <section> for the profile output. The output section contains three subsections that are revealed progressively using the CSS class visible, added and removed by JavaScript.

CSS Techniques

I used embedded CSS within a <style> block for all profile-specific styles. Key techniques include CSS transition on the .progress-bar width for smooth animation, @keyframes fadeIn and slideIn for revealing profile sections, and CSS Grid (flex-wrap) for the step indicator buttons. The display: none / display: block toggle pattern (via the .visible class) ensures sections are hidden by default and revealed by DOM manipulation.

JavaScript Implementation

The core JavaScript logic uses a profileData object to store answers across three steps, and a stepStatus object to track each step's state ('empty', 'partial', or 'full'). Key functions include:

  • startProfile() — finds the first incomplete step and calls runStep().
  • runStep(num) — iterates through each prompt in the step, stores answers in profileData, and computes the step status.
  • revisitStep(num) — allows the user to return to any step, offering to keep or overwrite existing answers.
  • updateProgressUI() — calculates percentage completion and updates the progress bar, label, and step buttons via DOM manipulation.
  • renderProfile() — calls renderStepSection() for each step, building table rows from stored data and toggling the .visible class on each section.

I also included an escapeHtml() utility to prevent cross-site scripting vulnerabilities when rendering user-provided input into the DOM.

Design Rationale

I chose a progress bar combined with colour-coded step buttons to give users immediate visual feedback on their completion state. The prompt()-based input was required by the specification; I mitigated its limited styling by wrapping it in clearly structured step logic with user-friendly messages. The profile output card uses the same card aesthetic as the global CSS to maintain visual consistency.

Sitemap Page — sitemap.html

View Live Page ↗ Validation Evidence ↗

HTML Structure

The page uses the shared template. The sitemap itself is embedded as an inline <svg> element inside a responsive wrapper <div class="svg-container">. Each page node is represented by an SVG <a> (anchor) element wrapping a <rect> (node box) and two <text> elements (page name and filename). Each <a> includes a <title> child for screen-reader accessibility and tabindex="0" for keyboard navigation.

SVG Techniques

The viewBox="0 0 860 540" attribute combined with width="100%" and no fixed height makes the SVG scale proportionally on all screen sizes. Connector lines are drawn with <line> elements using the CSS class connector. Node colour differentiation uses CSS fill classes (node-home, node-main, node-sub, etc.) which makes them easy to change globally.

CSS Hover & Focus

Hover effects are implemented entirely in CSS using .node-link:hover .node-rect selectors, changing the fill to the accent colour and increasing stroke-width. A :focus rule mirrors the hover styling and adds an outline for visible keyboard focus — satisfying WCAG 2.1 success criterion 2.4.7 (Focus Visible).

Design Rationale

I chose a top-down hierarchical layout to mirror a traditional website tree structure, which is intuitive for users. Colour coding differentiates student roles and shared pages, and a legend below the SVG explains the colour scheme. I avoided using bitmap images or screenshots, as required, building the entire diagram programmatically in SVG.

Content Page — content_ST4.html

View Live Page ↗ Validation Evidence ↗

HTML Structure

The content page uses a CSS Grid two-column layout (grid-template-columns: 220px 1fr) with a sticky <aside> sidebar for internal navigation, and an <article> as the main content container. Six <section> elements with unique id attributes allow the sidebar anchor links to function correctly. A scroll-margin-top offset compensates for the sticky header.

Internal Navigation

The sidebar contains anchor links using href="#section-id" pointing to each section's id. This satisfies the internal linking requirement without JavaScript. The "Go to Top" button uses href="#top" and is positioned with position: fixed, meaning it remains visible at all scroll positions.

Images

Four images sourced from Unsplash (royalty-free) are included, each with descriptive alt text, max-height, border-radius, and a box-shadow. The object-fit: cover property maintains consistent image height without distortion.

Design Rationale

The two-column layout with a sticky sidebar reflects common editorial and documentation patterns, improving usability for long-form content. I deliberately kept the typography clean and readable, with clear heading hierarchy from <h1> down to <h3>, supporting both human readability and accessibility tools.

Challenges & Lessons Learned

Challenge 1: JavaScript prompt() limitations

The User Profile specification requires data collection via prompt() rather than HTML forms. This is unusual for modern web development and presented challenges: prompts block the browser thread, cannot be styled, and behave differently across browsers. I resolved this by designing very clear, step-labelled prompt messages (e.g. "Step 1 of 3") and adding a confirm() dialog when revisiting already-answered fields, which gave users meaningful feedback without relying on custom UI elements.

Challenge 2: SVG responsiveness

Making the SVG sitemap scale well on small screens required careful use of viewBox and avoiding fixed pixel dimensions on the outer <svg> tag. I also found that CSS hover selectors on SVG <a> elements required descendant selectors (.node-link:hover .node-rect) rather than a simple :hover on the rect, which was a new technique I learned through experimentation.

Challenge 3: Template integration

Integrating my pages with Student 1's template and CSS required careful attention to avoid overriding global styles. I learned to prefix all my embedded CSS with specific class names (e.g. .profile-hero, .side-nav) to ensure no style leakage into shared components.

Lessons Learned

This project deepened my understanding of DOM manipulation, SVG authoring, and CSS architecture. I learned the importance of separating page-specific and global styles from the outset, and of designing for accessibility as a first-class concern rather than an afterthought.

Accessibility Considerations

Semantic HTML

All pages use semantic HTML5 elements: <header>, <nav>, <main>, <section>, <article>, <aside>, and <footer>. These landmarks are recognised by screen readers, allowing users to jump between page regions efficiently.

Images and alt text

Every image on all my pages includes a meaningful alt attribute that describes the image content in context. Decorative SVG elements (e.g. the site logo emoji) include aria-hidden="true" or a descriptive aria-label.

Keyboard navigation

All interactive elements are keyboard-accessible: buttons are native <button> elements, links are native <a> elements, and SVG nodes include tabindex="0". Focus styles are visible on all interactive elements, either inherited from browser defaults or explicitly defined in CSS (:focus { outline: 3px solid var(--accent); }).

ARIA attributes

The progress bar uses role="progressbar" with aria-valuenow, aria-valuemin, and aria-valuemax attributes that are updated dynamically by JavaScript. The profile output section uses aria-live="polite" so screen readers announce changes. The sitemap SVG includes a <title> and <desc> at the top level, and each node's <a> includes aria-label.

Colour contrast

All text-to-background colour combinations meet or exceed WCAG 2.1 AA contrast ratios (minimum 4.5:1 for normal text, 3:1 for large text). The primary green (#1a6b4a) on white achieves a ratio of approximately 7.2:1.
